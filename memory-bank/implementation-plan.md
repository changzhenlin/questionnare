# 实施计划（Implementation Plan）

**原则**：接口先行、模块化推进、每步可独立验收。

---

## Phase 1：项目脚手架与数据层

### 步骤 1.1：初始化 Next.js 15 项目
- **目标**：搭建 Next.js 15 + TypeScript + Tailwind CSS 基础工程，确认开发环境就绪。
- **验收标准**：
  - `pnpm dev` 成功启动，首页无报错。
  - 热更新（HMR）正常工作，修改文件后浏览器自动刷新。
- **测试要点**：访问 `http://localhost:3000`，确认页面渲染正常。

### 步骤 1.2：集成 shadcn/ui
- **目标**：初始化 shadcn/ui，安装 MVP 所需基础组件（Button、Input、Card、Dialog、Label、RadioGroup、Checkbox、Textarea）。
- **验收标准**：在任意页面引入上述组件，能正确渲染且样式一致，无全局样式污染。
- **测试要点**：创建一个测试页，逐个渲染组件，检查视觉和交互状态。

### 步骤 1.3：配置 Prisma 与数据库连接
- **目标**：安装 Prisma CLI 与 Client，配置数据库连接（开发环境使用本地 PostgreSQL 或 Vercel Postgres）。
- **验收标准**：
  - `prisma generate` 执行成功，生成类型安全的 Client。
  - `prisma db push` 能将 Schema 同步到数据库。
- **测试要点**：运行 Prisma Studio，确认可连接数据库并查看表结构。

### 步骤 1.4：设计数据库模型
- **目标**：定义问卷（Survey）、题目（Question）、选项（Option）、答卷（Response）、答案（Answer）五张核心表的关系模型，明确字段、类型、约束、级联行为。
- **验收标准**：
  - Schema 通过 `prisma validate` 无报错。
  - 关系定义正确：问卷 1:N 题目、题目 1:N 选项、问卷 1:N 答卷、答卷 1:N 答案。
  - 删除问卷时级联删除其题目、选项、答卷、答案。
- **测试要点**：
  - 检查外键关系与 `onDelete: Cascade` 配置。
  - 确认用户表由 Clerk 管理，业务表通过 `userId`（字符串）与用户关联。

### 步骤 1.5：集成 Clerk 认证
- **目标**：配置 Clerk，实现注册、登录、登出、会话保护。
- **验收标准**：
  - 未登录用户访问管理页面被重定向到登录页。
  - 登录后 Session 持久化，刷新页面保持登录态。
  - 登出后清除 Session，返回首页。
- **测试要点**：
  - 登录态下访问 `/dashboard` 正常进入。
  - 清除 Cookie 后访问受保护路由被拦截。

## Phase 2：接口层先行 — Schema 与 API 契约

> **本阶段原则**：先定义好前后端交互的数据结构和接口契约，前端页面在 Mock 数据下即可并行开发。

### 步骤 2.1：定义 Zod 校验 Schema
- **目标**：为所有 API 的请求体和响应体定义 Zod Schema，前后端共用同一套校验规则。
- **验收标准**：
  - 覆盖问卷创建、更新、发布/结束、答卷提交的请求校验。
  - 覆盖题目（单选/多选/填空）及选项的数据结构校验。
  - 非法数据传入时，校验失败并返回结构化错误信息。
- **测试要点**：
  - 传入空标题、空选项、非法题型枚举值，校验应失败。
  - 传入合法数据，校验通过。

### 步骤 2.2：实现问卷 CRUD API
- **目标**：实现问卷的创建、读取、更新、删除四个核心接口。
- **验收标准**：
  - `POST /api/surveys`：创建问卷，返回新问卷 ID。
  - `GET /api/surveys`：返回当前用户问卷列表（含分页）。
  - `GET /api/surveys/:id`：返回问卷详情（含题目、选项）。
  - `PUT /api/surveys/:id`：更新问卷标题、描述、题目内容。
  - `DELETE /api/surveys/:id`：删除问卷并级联清理关联数据。
  - 所有写操作校验用户所有权，越权访问返回 403。
- **测试要点**：
  - 用户 A 无法通过 API 操作用户 B 的问卷。
  - 更新问卷后，再次读取返回最新数据。
  - 删除问卷后，关联题目、选项、答卷、答案全部被清理。

### 步骤 2.3：实现问卷发布/结束/重新发布 API
- **目标**：实现问卷状态流转接口：草稿 ↔ 已发布 ↔ 已结束。
- **验收标准**：
  - `POST /api/surveys/:id/publish`：草稿问卷发布，校验至少包含一个题目。
  - `POST /api/surveys/:id/close`：已发布问卷结束。
  - `POST /api/surveys/:id/reopen`：已结束问卷重新发布。
  - 非法状态流转返回 400。
- **测试要点**：
  - 空题目问卷不能发布。
  - 已结束问卷无法直接编辑，但可重新发布。

### 步骤 2.4：实现答卷提交 API
- **目标**：实现公开访问的答卷提交接口，接收问卷 ID 和答案数组。
- **验收标准**：
  - `POST /api/surveys/:id/submit`：接收答案，保存答卷和答案记录。
  - 已结束或草稿问卷拒绝提交，返回 400。
  - 必填题目未答返回 400 及具体错误信息。
  - 单选答案必须是合法选项 ID；多选答案必须是选项 ID 数组；填空答案必须是字符串。
- **测试要点**：
  - 提交合法答卷后，数据库 `Response` 和 `Answer` 表新增记录。
  - 提交已结束问卷，返回错误且数据库无新增记录。
  - 单选题提交非法选项 ID，返回校验错误。

### 步骤 2.5：实现数据统计 API
- **目标**：实现按问卷 ID 聚合统计数据的只读接口。
- **验收标准**：
  - `GET /api/surveys/:id/stats`：返回总回收数、浏览数。
  - 单选题：返回每个选项的计数和百分比。
  - 多选题：返回每个选项的计数（百分比分母为答卷总数）。
  - 填空题：返回答案列表（支持分页）。
  - 只有问卷所有者能访问统计接口，越权返回 403。
- **测试要点**：
  - 提交多份答卷后，统计数字与数据库记录一致。
  - 空问卷（无答卷）返回统计空状态，不报错。

## Phase 3：问卷管理前端（用户侧）

> **开发方式**：基于 Phase 2 的 API 契约，前端可用 Mock 数据并行开发，接口就绪后直接替换。

### 步骤 3.1：问卷列表页（Dashboard）
- **目标**：展示当前用户所有问卷，显示标题、状态、创建时间，支持按状态筛选。
- **验收标准**：
  - 页面加载后正确渲染问卷列表。
  - 支持按"全部/草稿/已发布/已结束"筛选。
  - 空列表时显示友好引导（如"创建你的第一份问卷"）。
  - 点击问卷进入编辑页，点击"查看统计"进入结果页。
- **测试要点**：
  - 创建新问卷后，列表自动刷新显示新记录。
  - 筛选状态切换后，列表内容正确过滤。

### 步骤 3.2：问卷编辑器 — 题目增删改
- **目标**：实现问卷编辑页，支持添加题目（单选/多选/填空）、编辑题目文本、添加/删除选项。
- **验收标准**：
  - 可添加三种题型题目，每种题目可输入题干。
  - 单选/多选题可动态添加、删除选项。
  - 可删除已添加的题目。
  - 保存后数据通过 API 持久化，刷新页面正确回填。
- **测试要点**：
  - 题目为空或选项为空时保存，前端给出校验提示。
  - 保存后进入列表页，再返回编辑页，数据完整回填。

### 步骤 3.3：问卷编辑器 — 题目排序
- **目标**：支持拖拽或按钮调整题目顺序。
- **验收标准**：
  - 用户可通过拖拽或上下按钮改变题目顺序。
  - 顺序变更后保存，再次加载时顺序一致。
- **测试要点**：调整顺序后保存，刷新页面验证顺序未错乱。

### 步骤 3.4：问卷发布与分享
- **目标**：在编辑页或列表页提供发布、结束、重新发布操作，发布后展示分享链接。
- **验收标准**：
  - 草稿问卷点击"发布"后状态变为已发布，生成公开链接。
  - 点击"复制链接"将分享 URL 写入剪贴板。
  - 已发布问卷可"结束"，结束后链接不可再填写。
  - 已结束问卷可"重新发布"。
- **测试要点**：
  - 发布后复制链接，在无痕窗口打开可正常显示问卷。
  - 结束后访问链接，显示"问卷已结束"提示。

## Phase 4：问卷填写前端（访客侧）

### 步骤 4.1：问卷填写页
- **目标**：公开访问的问卷填写页面，根据 URL 中的问卷 ID 拉取问卷数据并渲染。
- **验收标准**：
  - 正确渲染问卷标题、描述、所有题目和选项。
  - 单选使用 Radio，多选使用 Checkbox，填空使用 Input/Textarea。
  - 必填题目标注必填标识，未填提交时给出提示。
  - 提交成功后显示感谢页面或成功提示。
  - 问卷为草稿或已结束时，显示对应状态提示（不可填写）。
- **测试要点**：
  - 单选/多选/填空交互正常，表单数据正确收集。
  - 提交后数据库生成答卷记录。
  - 已结束问卷访问时，不渲染表单，仅展示结束提示。

## Phase 5：数据统计与可视化

### 步骤 5.1：统计概览页
- **目标**：展示问卷回收概况与各题目统计结果。
- **验收标准**：
  - 页面顶部显示总回收数、浏览数。
  - 单选题/多选题以柱状图展示各选项计数。
  - 填空题以列表展示答案（分页加载）。
  - 无数据时显示空状态，不报错。
- **测试要点**：
  - 提交答卷后刷新统计页，数字与图表正确更新。
  - 多选题柱状图计数不重复计算（一张答卷选多个选项时，每个选项各计一次）。

### 步骤 5.2：答卷详情列表
- **目标**：展示该问卷的所有答卷，支持逐份查看每题答案。
- **验收标准**：
  - 按提交时间倒序展示答卷列表。
  - 点击单份答卷可查看该答卷每题的具体答案。
  - 分页加载，避免大数据量卡顿。
- **测试要点**：
  - 答卷总数与统计页回收数一致。
  - 单份答卷的答案与填写时选择一致。

## Phase 6：质量保障与优化

### 步骤 6.1：响应式适配
- **目标**：确保问卷填写页和管理页在移动端可用。
- **验收标准**：
  - iPhone SE（375px）至桌面端（1440px）无布局错乱。
  - 触控目标（按钮、选项）最小 44×44px，易于点击。
- **测试要点**：
  - Chrome DevTools 模拟 iPhone 14 Pro，完整走通"创建→发布→填写→查看统计"流程。

### 步骤 6.2：核心流程 E2E 测试
- **目标**：覆盖用户核心使用路径的端到端测试。
- **验收标准**：
  - 测试覆盖完整主流程：注册登录 → 创建问卷 → 添加单选/多选/填空题 → 发布 → 复制链接 → 无痕窗口填写 → 查看统计。
  - 每个断言验证页面 UI 状态和数据库状态一致。
- **测试要点**：
  - 使用 Playwright 或 Cypress 编写自动化测试。
  - 确保测试间数据隔离（每个测试创建独立问卷）。

### 步骤 6.3：性能与安全基线检查
- **目标**：满足 PRD 非功能需求。
- **验收标准**：
  - 首屏加载时间 ≤ 2s（Lighthouse Performance ≥ 80）。
  - 所有写接口校验用户身份，防止越权操作。
  - 表单输入防止 XSS（对富文本进行转义，本项目无富文本输入则确保纯文本存储）。
  - API 路由开启基础 Rate Limit（如每 IP 每秒最多 10 次提交）。
- **测试要点**：
  - Lighthouse 跑分检查。
  - 尝试直接调用 API 修改他人问卷，验证返回 403。
  - 快速连续点击提交按钮，验证限流生效。

## 模块化边界说明

- **认证模块（Clerk）**：负责登录态校验，业务代码只消费 `userId`，不处理密码或 Token。
- **数据校验模块（Zod）**：前后端共用同一套 Schema，接口层统一校验，不分散到组件内部。
- **API 层**：每个资源（Survey、Response、Stats）独立路由文件，互不依赖内部实现。
- **前端页面层**：管理页（`/dashboard`、 `/survey/:id/edit`）与访客页（`/survey/:id`）完全分离，复用组件通过 props 传入差异配置，不用条件分支判断环境。
